Tier 1 · same pipeline, two corpora

Same chunking. Same embedder. Different loaders.

langchain-oracledb doesn't care whether your text came from PDFs on disk or HTML over the wire. The loader on the left changes; the storage and retrieval on the right don't.

  • PyPDFLoader — page-aware chunks with source.pdf:p.N citations
  • WebBaseLoader — HTML scrubbed via trafilatura, URL kept as metadata
  • Same OracleVS.from_documents() sink either way
  • Mix-and-match: PDFs and web pages in the same collection
from langchain_community.document_loaders import (
    PyPDFLoader, WebBaseLoader,
)
from langchain_oracledb import OracleVS

pdf_docs = PyPDFLoader("corpus/release-notes.pdf").load()
web_docs = WebBaseLoader([
    "https://docs.oracle.com/en/database/.../vector.html",
]).load()

vs = OracleVS.from_documents(
    pdf_docs + web_docs,         # mixed corpus, same sink
    embedding=embedder,
    client=conn,
    table_name="CORPUS",
    distance_strategy="COSINE",
)

The corpus shape changes; the retrieval path doesn't. Beginner badges 1.1 (PDFs) and 1.3 (web articles) ship the same vs object — just a different load() at the top.

Tier 2 · Intermediate

Oracle MCP — the model calls tools, not schemas.

Without MCP, you stuff the schema into the system prompt — ~600 tokens of CREATE TABLEs and indexes, and you pray the model doesn't hallucinate a column. Cost climbs with every table; reliability drops with every join.

With Oracle MCP, the model has four tools and zero schema in the prompt. It picks list_tables when it needs the surface, describe_table when it needs columns, run_sql when it has the query, vector_search when it has only a question.

# Oracle MCP server exposes four tools.
# The agent picks; Oracle answers.

tools = [
    ListTables(),                          # schema discovery
    DescribeTable(),                       # columns + indexes
    RunSQL(mode="read_only"),              # SELECTs by default
    VectorSearch(model="MY_MINILM_V1"),    # in-DB ONNX
]

agent = create_react_agent(
    llm=ChatOpenAI(model="xai.grok-4"),
    tools=tools,
)

# The agent doesn't memorize your schema. It asks.

Bearer-token auth still applies — the same OCI_GENAI_API_KEY env var from slide 5 picks up here. The difference is what the model sees: in slide 5 it sees text, in slide 8 it sees tools.

Title Promise Stack Beginner Intermediate
Tier 2 · Intermediate

Stop sending schemas to the model. Let it call tools instead.

A LangChain tool-calling agent with four locally-defined BaseTool subclasses — the MCP shape without the separate-server overhead. Grok 4 picks which to call, run by run.

  • list_tables() — read-only schema discovery
  • describe_table(name) — column types, indexes
  • run_sql(query) — read-only by default; explicit opt-in for writes
  • vector_search(text, k) — over in-DB embeddings

Every SQL the agent runs is teed to logs/agent.sql.log via SQLcl — so you can replay, audit, or paste into the docs after the demo.

from langchain_core.tools import BaseTool
from langchain_openai import ChatOpenAI

tools = [ListTables(), DescribeTable(),
         RunSQL(mode="read_only"),
         VectorSearch(model="MY_MINILM_V1")]

llm = ChatOpenAI(
    model="xai.grok-4",
    base_url=BEARER_URL,
    api_key=os.environ["OCI_GENAI_API_KEY"],
).bind_tools(tools)

# Agent loop: LLM picks tool → SQLcl tees → result → LLM

Same protocol · your model · your hardware

Same protocol. Your model on your hardware.

MCP isn't OCI-specific. The same Oracle MCP server from slide 9 speaks to Ollama running a local model — the four tools (list_tables, describe_table, run_sql, vector_search) don't change.

  • ollama serve on :11434 — OpenAI-compat endpoint
  • Qwen3.5-35B-A3B at INT4 — fits a 24 GB A10
  • MCP client config points at the same Oracle MCP server
  • Falls back to Grok-4 over OCI when local hardware is unavailable

Caveat. Tool-call quality tracks model quality. Qwen3.5-35B-A3B is the floor we recommend; smaller models will hallucinate tool arguments under load.

from langchain_openai import ChatOpenAI

# Local Ollama, OpenAI-compatible endpoint
llm = ChatOpenAI(
    model="qwen3.5:35b-a3b",
    base_url="http://localhost:11434/v1",
    api_key="ollama",         # any non-empty string
).bind_tools(oracle_mcp_tools)

# Same agent loop. Same Oracle MCP server.
# Different inference plane.
agent = create_react_agent(llm, tools=oracle_mcp_tools)

Title Promise Stack Intermediate Advanced
Embeddings live inside Oracle

Same model. No Python embedder process.

Most projects ship a separate Python embedder process: a model file on disk, a HuggingFace import, its own failure mode. onnx2oracle registers the same MiniLM-L6-v2 ONNX model inside Oracle so embedding becomes a SQL primitive — same connection as the rest of your retrieval path.

  • Register once via the onnx2oracle CLI
  • One round-trip: query in, vector out, all in-database
  • Embeddings + vector search + relational filters in one SQL plan
  • OracleVS and oracleagentmemory share one embedder, one schema, one operational story
# 1. Register the ONNX model (one-time)
$ onnx2oracle export --preset all-MiniLM-L6-v2 \
                     --output minilm.onnx
$ onnx2oracle import \
    --dsn user/pwd@host:port/service \
    --onnx minilm.onnx \
    --model-name MY_MINILM_V1

# 2. Embed and search in one SQL pass
SELECT id, text,
       VECTOR_DISTANCE(
         embedding,
         VECTOR_EMBEDDING(
           MY_MINILM_V1 USING :q AS data
         ),
         COSINE
       ) AS dist
FROM   chunks
ORDER BY dist
FETCH FIRST 4 ROWS ONLY;

Beyond single-pipeline RAG

Retrieval. Reasoning. Tool use. Three agents, one query.

apps/agentic_rag/ splits the work that beginner-tier RAG does in one chain across three specialised agents. Each agent owns one decision; the orchestrator (LangGraph) wires them; oracleagentmemory carries the conversation state across them.

  • Retrieval agent. Picks the collection (RUNBOOKS / GLOSSARY / DECISIONS), runs vector search, ranks results.
  • Reasoning agent. Plans the answer over the retrieved context using CoT or ReAct.
  • Tool agent. Calls Oracle MCP if SQL or fresh-data lookup is needed mid-answer.
from langgraph.graph import StateGraph

graph = StateGraph(AgentState)
graph.add_node("retrieve", retrieval_agent)
graph.add_node("reason",   reasoning_agent)
graph.add_node("tools",    tool_agent)

graph.add_edge("retrieve", "reason")
graph.add_conditional_edges("reason",
    route=lambda s: "tools" if s.needs_sql else END,
)

# Conversation state via OAMP, not in-process memory
thread = client.create_thread(user_id, agent_id)

Big teacher · small student · traces in between

Big teacher, small student — and the traces in between.

When the deployed model has to be small (latency, cost, on-device) but the answers should look big-model-shaped, distillation moves the reasoning capacity instead of the weights.

  • Teacher. Qwen3.5-35B-A3B or Grok-4 generates a chain-of-thought reasoning trace per question.
  • Trace splitter. split_reasoning.py segments each trace into (prompt, reasoning, answer) triples.
  • Student. Smaller Qwen (e.g. 7B) is fine-tuned on the triples — supervised, no RL.
  • Result. The student produces teacher-shaped reasoning at student-shaped cost.
# 1. Teacher generates reasoning traces
$ python apps/agent-reasoning/generate_reasoning_outputs.py \
    --model qwen3.5:35b-a3b \
    --questions data/questions.jsonl \
    --out traces/raw.jsonl

# 2. Split each trace into (prompt, reasoning, answer)
$ python apps/agent-reasoning/split_reasoning.py \
    --in  traces/raw.jsonl \
    --out traces/triples.jsonl

# 3. Fine-tune the student on the triples
$ trl sft --model qwen3.5:7b \
          --dataset traces/triples.jsonl \
          --output_dir student-7b-distilled

Oracle isn't shipping this pipeline — but it's how every workshop attendee should be thinking about deploying small models locally on the same A10 box.

Self-memory across processes

Cold → kill → warm. The agent remembers.

In the v2 walk-through, an agent fetched the AI Vector Search guide, summarised it, and stored the summary in a JSON Duality View. We killed the FastAPI process. A fresh process — different PID, empty Python heap — answered a follow-up question with the stored text verbatim.

No vector cache replay. No serialised chain state. Just SQL — or, in v3, just oracleagentmemory.get_thread(saved_id) over the same SQL.

# Process A — store
view.insert({
  "agent_id": "researcher-01",
  "memory_type": "semantic",
  "text": summary,
  "source_url": doc.url,
  "embedded_at": now(),
})
# <Ctrl-C the FastAPI server>

# Process B — fresh PID, empty heap
hits = view.vector_search(
  query="How does Oracle implement HNSW?",
  k=1, agent_id="researcher-01",
)
assert hits[0].text in answer  # PASS
